## 2. 缺陷状态定义

| 状态 | 定义 | 操作者 |
|------|------|--------|
| New (新建) | 缺陷刚创建，待评审 | 测试人员 |
| Triage (评审) | 缺陷已评审，确定有效，待分配 | Test Lead |
| Assigned (已分配) | 缺陷已分配给开发人员 | Test Lead |
| In Progress (处理中) | 开发人员正在修复 | 开发人员 |
| Fixed (已修复) | 代码已修复，待测试验证 | 开发人员 |
| Verified (已验证) | 修复已验证通过 | 测试人员 |
| Reopened (重新打开) | 验证失败，重新打开 | 测试人员 |
| Closed (已关闭) | 缺陷修复完成，已验证 | Test Lead |
| Won't Fix (不修复) | 确认不修复，关闭 | Test Lead |
| Cannot Repro (无法复现) | 无法复现，关闭 | Test Lead |

## 3. 缺陷等级定义

| 等级 | 定义 | 示例 | 修复时限 |
|------|------|------|----------|
| S1-Critical (致命) | 导致系统崩溃、数据丢失、安全问题 | 系统死机、财产损失 | 24h |
| S2-Major (严重) | 功能失效，影响核心功能 | 解锁失败、无法启动 | 72h |
| S3-Minor (一般) | 功能异常，但有替代方案 | 响应慢、显示错 | 2周 |
| S4-Trivial (提示) | 轻微问题，不影响使用 | 文字错位、UI细节 | 下一版本 |

## 5. 缺陷处理流程

### 5.1 创建缺陷

1. 测试人员发现缺陷
2. 记录缺陷信息到缺陷跟踪系统
3. 标记状态为 New
4. 附上复现步骤、日志、截图

### 5.2 评审缺陷

1. Test Lead 评审缺陷有效性
2. 确认缺陷是否有效
3. 确定优先级和严重度
4. 指派给开发人员
5. 标记状态为 Assigned

### 5.3 修复缺陷

1. 开发人员接收缺陷
2. 分析缺陷原因
3. 进行代码修复
4. 提交修复代码
5. 标记状态为 Fixed
6. 填写修复说明

### 5.4 验证缺陷

1. 测试人员获取修复版本
2. 按照复现步骤验证
3. 验证通过 → 标记 Verified
4. 验证失败 → 标记 Reopened，返回开发

### 5.5 关闭缺陷

1. Test Lead 审查修复结果
2. 确认修复完整
3. 标记状态为 Closed

## 7. 缺陷统计指标

| 指标 | 计算方法 | 目标 |
|------|----------|------|
| 缺陷密度 | 缺陷数/KLOC | <1.0 |
| P0缺陷数 | S1缺陷计数 | 0 |
| 平均修复时间 | 修复时间求平均 | <72h |
| 缺陷逃逸率 | 上线后缺陷数/总缺陷数 | <5% |
| 重测通过率 | 一次通过数/总数 | >90% |
