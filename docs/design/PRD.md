## 1. 产品概述与背景

### 1.1 产品定位

数字钥匙系统（Digital Key System）是面向智能网联汽车的新一代无钥匙进入与启动解决方案，通过将实体车钥匙功能数字化迁移至智能手机及可穿戴设备，实现"手机即钥匙"的创新体验。系统同时兼容国内外两大主流标准（ICCE 与 CCC），支持多车型适配与多用户共享，构建完整的数字钥匙生态。

### 1.2 市场背景

随着智能网联汽车的快速普及，传统物理钥匙的使用痛点日益突出：忘带钥匙、钥匙丢失、钥匙外借不便等问题严重影响用户体验。与此同时，智能手机的算力提升、NFC/UWB/BLE 等近场通信技术的成熟，以及车主对智能化服务的需求增长，共同推动了数字钥匙市场的爆发式增长。

数字钥匙不仅是物理钥匙的替代品，更是车联网生态的核心入口——未来可承载车辆共享、分时租赁、UBI 保险、汽车融资租赁等多元化商业模式。

### 1.3 业务目标

| 目标维度 | 量化指标 |
|---------|---------|
| 协议兼容 | 同时支持 ICCE 与 CCC 两大标准，实现国内外全车型覆盖 |
| 用户体验 | 车门解锁响应时间 < 1 秒；离线可用；多车辆统一管理 |
| 安全保障 | 满足 EAL5+ 安全等级，端到端加密，防中继攻击 |
| 扩展能力 | 支持百万级用户接入，提供标准化 SDK 供第三方集成 |
| 合规要求 | 符合 T/CA 110-2020（ICCE）及 CCC Digital Key 3.0 规范 |

### 1.4 术语表

| 术语 | 定义 |
|-----|------|
| ICCE | 智慧车联产业生态联盟（Intelligent Car Connectivity Ecosystem），中国本土数字钥匙标准组织 |
| CCC | Car Connectivity Consortium，国际汽车连接联盟，制定 CCC Digital Key 规范 |
| UWB | Ultra-Wideband，超宽带无线通信技术，用于厘米级精准测距 |
| BLE | Bluetooth Low Energy，蓝牙低功耗，用于近场通信与连接管理 |
| NFC | Near Field Communication，近场通信，支持被动式快速刷卡解锁 |
| SE | Secure Element，安全元件，提供硬件级密钥存储与加密运算 |
| TEE | Trusted Execution Environment，可信执行环境，提供安全隔离的执行环境 |
| PKI | Public Key Infrastructure，公钥基础设施，用于身份认证与密钥分发 |
| eSE | embedded Secure Element，嵌入式安全芯片 |
| eUICC | embedded Universal Integrated Circuit Card，嵌入式 SIM |

## 2. 用户角色定义

### 2.1 角色总览

```
车主（Owner）
  ├── 主账号车主（Primary Owner）
  └── 家庭成员（Family Member）
授权用户（Authorized User）
  ├── 朋友/同事（Friend/Colleague）
  ├── 代驾司机（Valet Driver）
  └── 租车用户（Rental User）
车厂管理员（OEM Admin）
服务提供商（Service Provider）
  ├── 经销商（Dealer）
  ├── 保险公司（Insurance Company）
  └── 出行平台（ Mobility Platform）
运维人员（Operations）
```

### 2.2 角色详细定义

#### 2.2.1 主账号车主（Primary Owner）

**定义：** 车辆购买者或主要使用者，拥有数字钥匙的最高管理权限。

**权限范围：**
- 创建、分享、吊销所有类型的数字钥匙
- 绑定/解绑车辆与用户账号
- 管理车辆安全设置（允许他人启动、速度限制等）
- 查看所有钥匙使用记录与审计日志
- 转让车辆所有权（包含钥匙权限转移）

**使用场景：**
- 日常通勤：携带手机靠近车辆即解锁，离开后自动上锁
- 钥匙分享：临时授权给家人或朋友，无需交接物理钥匙
- 远程控车：高温天提前启动空调，查看车辆状态

#### 2.2.2 家庭成员（Family Member）

**定义：** 与主车主在同一家庭账号体系下的成员。

**权限范围：**
- 使用数字钥匙解锁与启动车辆
- 查看车辆状态（部分信息）
- 无法分享或吊销钥匙
- 无权管理车辆安全设置

**使用场景：**
- 家庭成员无需额外申请，即可使用已分享的主钥匙或副钥匙

#### 2.2.3 临时授权用户（Temporary User）

**定义：** 通过分享链接或临时钥匙获得车辆有限使用权的用户。

**权限范围：**
- 在指定时间窗口内解锁与启动车辆
- 超出时间窗口后钥匙自动失效
- 可设置地理围栏限制使用区域
- 无法查看车辆敏感数据

**使用场景：**
- 代驾场景：预约代驾司机，授权其 2 小时内的车辆使用权限
- 修车场景：授权维修人员将车开至 4S 店

#### 2.2.4 车厂管理员（OEM Admin）

**定义：** 汽车制造商的系统管理员，负责数字钥匙后台管理。

**权限范围：**
- 开通/关闭特定车型的数字钥匙服务
- 管理车企密钥（Vehicle Master Key）
- 查看辖下车型的使用统计数据
- 配置 OTA 推送策略

#### 2.2.5 服务提供商（Service Provider）

**定义：** 通过 SDK/API 接入数字钥匙生态的第三方合作伙伴。

**权限范围：**
- 通过标准化 API 管理其业务范围内的数字钥匙
- 接收钥匙事件通知（如车辆解锁、启动）
- 无法访问车辆控制权限之外的数据

## 3. 核心功能详细设计

### 3.1 嵌入式端（车端）

#### 3.1.1 硬件架构

车端数字钥匙系统由以下硬件模块构成：

```
┌─────────────────────────────────────────────────────────┐
│                    中央安全网关（Central Gateway）        │
│  ┌──────────────┐  ┌──────────────┐  ┌──────────────┐  │
│  │   SE 芯片    │  │  协议栈固件   │  │  OTA 引擎    │  │
│  │  (EAL5+)     │  │  (ICCE+CCC)  │  │              │  │
│  └──────────────┘  └──────────────┘  └──────────────┘  │
├─────────────────────────────────────────────────────────┤
│  通信模块（Communication Module）                        │
│  ┌────────┐  ┌────────┐  ┌────────┐                    │
│  │  UWB   │  │  BLE   │  │  NFC   │                    │
│  │  模块   │  │  模块   │  │  模块   │                    │
│  └────────┘  └────────┘  └────────┘                    │
├─────────────────────────────────────────────────────────┤
│  定位天线阵列（UWB Antenna Array）—— 通常 4-6 根天线      │
│  BLE/NFC 天线                                           │
└─────────────────────────────────────────────────────────┘
```

#### 3.1.2 通信模块功能设计

**UWB 模块（超宽带）**

| 功能项 | 规格说明 |
|-------|---------|
| 测距精度 | ≤ 10cm（开阔环境）；≤ 30cm（复杂车内环境） |
| 定位精度 | 厘米级三维定位（结合多天线阵列） |
| 测距范围 | 有效范围 0.5m ~ 10m（可配置） |
| 通信速率 | 最大 110 Mbps（IEEE 802.15.4z） |
| 安全测距 | 符合 IEEE 802.15.4z HRP UWB 规范，支持 Secure Ranging |
| 抗干扰 | 支持频段 3.1 GHz ~ 10.6 GHz，多信道跳频 |
| 功耗 | 主动测距时 < 200mW |
| 唤醒机制 | 支持 BLE 辅助唤醒，节省整车功耗 |

**BLE 模块（蓝牙低功耗）**

| 功能项 | 规格说明 |
|-------|---------|
| BLE 版本 | Bluetooth 5.0 及以上 |
| 角色支持 | Central（车端作为外设，即 Peripheral） |
| 连接参数 | 连接间隔 30ms~50ms，保活周期 30s |
| GATT 服务 | 标准 CCC/ICCE GATT Profile |
| 安全模式 | LE Secure Connections (LE SC)，支持 Secure Authenticated Link |
| 广播 | 可配置广播间隔 100ms~1s，支持扩展广播 |
| 配对绑定 | 支持 Passkey Entry、Numeric Comparison 配对方式 |
| 多设备 | 支持同时与最多 8 台设备保持 BLE 连接 |

**NFC 模块**

| 功能项 | 规格说明 |
|-------|---------|
| NFC 协议 | ISO/IEC 14443 Type A/B、ISO/IEC 18092（FeliCa） |
| 通信距离 | 0 ~ 4cm（被动式刷卡） |
| 传输速率 | 106/212/424 kbps |
| 安全通道 | 支持 ISO/IEC 7816-4 APDU 安全指令 |
| 离线支持 | NFC 刷卡解锁不依赖手机电量（手机没电时可用） |
| 刷卡区域 | 通常位于车门把手或中控台，位置标准化 |

#### 3.1.4 安全子系统设计

**安全芯片（SE）集成**

| 安全功能 | 实现要求 |
|---------|---------|
| 密钥存储 | 主密钥（Master Key）、会话密钥（Session Key）全部存储在 SE 芯片内 |
| 加密运算 | 支持 SM2/SM3/SM4（ICCE）和 ECDSA/P-256/AES-256（CCC） |
| 认证等级 | EAL5+ 及以上 |
| 防篡改 | 硬件级防物理/逻辑攻击 |
| 密钥生命周期 | 支持密钥安全销毁、版本回滚保护 |
| 安全启动 | 系统启动时对固件进行完整性校验 |
| 侧信道防护 | 抗时序攻击、功耗分析攻击 |

**防中继攻击（Relay Attack Protection）**

| 防护机制 | 实现方式 |
|---------|---------|
| UWB 物理层测距 | 利用 UWB 信号飞行时间（ToF）测量真实距离，确保手机在车辆有效范围内 |
| 时间窗口校验 | ICCE/CCC 协议均定义最大允许时延（~3μs），超出则判定为中继攻击 |
| 距离边界检测 | 解锁指令仅在 UWB 测量距离 < 设定阈值（如 2m）时执行 |
| 加密通道 | 所有测距数据在加密通道中传输，防止篡改 |
| 随机数挑战 | 每次测距使用一次性随机数（Nonce），防止重放 |

#### 3.1.5 钥匙生命周期管理（车端）

```
创建 → 预配对 → 配对完成 → 激活 → 正常使用 → 更新 → 吊销 → 删除
  │        │          │         │       │       │      │      │
  └────────┴──────────┴─────────┴───────┴───────┴──────┴──────┘
                    （完整生命周期状态机）
```

**状态说明：**
- **创建（Created）：** 云端生成钥匙模板，推送至车端
- **预配对（Pre-Paired）：** 车端接收并暂存钥匙公钥信息，等待手机侧配对
- **配对完成（Paired）：** 双向身份认证成功，共享密钥建立
- **激活（Active）：** 钥匙生效，可正常使用
- **更新（Updated）：** 密钥轮换或权限变更
- **吊销（Revoked）：** 钥匙被远程禁用
- **删除（Deleted）：** 钥匙记录从车端完全清除

#### 3.2.2 核心功能模块

**模块一：钥匙创建与注册**

| 功能 | 详细描述 |
|-----|---------|
| 车辆发现 | 通过扫码（车端显示配对码）、蓝牙扫描、NFC 触碰三种方式发现车辆 |
| 身份认证 | 支持手机账号登录、人脸识别、指纹识别等生物认证方式 |
| 车主证明 | 调用车厂 API 验证车辆所有权（VIN 码 + 购车证明） |
| 钥匙生成 | 在 SE/TEE 安全环境中生成非对称密钥对，公钥上传云端 |
| 协议选择 | 自动识别车辆支持 ICCE/CCC 协议，选用对应协议完成注册 |
| OTA 激活 | 钥匙创建指令通过 OTA 下发至车端，完成双向绑定 |

**模块二：钥匙分享**

| 功能 | 详细描述 |
|-----|---------|
| 分享方式 | 二维码分享、链接分享、NFC 碰一碰分享 |
| 权限级别 | 主钥匙（Full Access）、副钥匙（Limited Admin）、临时钥匙（Time/Location Restricted） |
| 时间控制 | 支持设定生效时间、失效时间、有效天数 |
| 地理围栏 | 支持设定可用城市/区域，超出范围钥匙自动锁定 |
| 使用次数 | 支持设定钥匙可用次数（如：仅允许解锁 5 次） |
| 实时撤销 | 分享者可随时在 App 内撤销分享的钥匙 |
| 分享记录 | 完整记录每次分享的时间、对象和使用情况 |

**模块三：远程控车**

| 功能 | ICCE | CCC | 说明 |
|-----|------|-----|------|
| 远程解锁 | ✓ | ✓ | 通过云端下发解锁指令至车端 |
| 远程上锁 | ✓ | ✓ | 通过云端下发上锁指令至车端 |
| 远程启动 | ✓ | ✓ | 发动机启动（部分国家法规限制） |
| 闪灯鸣笛 | ✓ | ✓ | 寻车功能 |
| 升降车窗 | ✓ | ✓ | 控制车窗、天窗开闭 |
| 空调控制 | ✓ | ✓ | 预设温度、座椅加热等 |
| 车辆状态 | ✓ | ✓ | 实时查看车门、车窗、电量等状态 |

**模块四：多车辆管理**

| 功能 | 详细描述 |
|-----|---------|
| 车辆列表 | 支持同时管理多辆车，车辆卡片展示车牌、品牌、型号 |
| 快速切换 | 首页常驻当前使用车辆，支持一键切换 |
| 统一入口 | 所有车辆钥匙统一在一个 App 内管理 |
| 车辆排序 | 支持按添加时间、常用程度、最近使用自动排序 |

**模块五：离线模式**

| 功能 | 详细描述 |
|-----|---------|
| NFC 刷卡 | 手机没电或无网络时，通过 NFC 仍可解锁车辆 |
| 离线钥匙 | 在无网络环境下，预下载的钥匙授权数据仍可使用 |
| 延迟同步 | 离线期间的操作记录，待网络恢复后自动同步至云端 |
| 有效期保障 | 离线钥匙在有效期内持续有效，过期自动失效 |

### 3.3 云端（服务平台）

#### 3.3.1 系统架构

```
┌─────────────────────────────────────────────────────────────────┐
│                         接入层（API Gateway）                     │
```

## 5. ICCE 协议与 CCC 协议功能映射

### 5.1 两大协议对比总览

| 维度 | ICCE（中国标准） | CCC（国际标准） |
|-----|-----------------|----------------|
| 制定组织 | 智慧车联产业生态联盟 | Car Connectivity Consortium |
| 最新版本 | T/CA 110-2020 系列 | CCC Digital Key 3.0 |
| 协议架构 | 四层架构（应用/传输/适配/物理） | 六层架构（Secure Domain/VCI/等） |
| 核心算法 | 国密算法（SM2/SM3/SM4） | 国际算法（ECDSA/P-256/AES-256） |
| 通信模态 | UWB + BLE + NFC | UWB + BLE + NFC（全模） |
| 安全芯片 | 支持 SE（国密） | 支持 SE（国际标准） + Secure Domain |
| 认证体系 | 国密认证 | FIDO / GSMA SAS |
| 生态覆盖 | 国内车企为主 | 全球车企（Apple/Google/Samsung） |
| 互操作性 | 国内车企内部互认 | 全球 CCC 成员互认 |

### 5.2 功能映射矩阵

| 功能模块 | ICCE 实现 | CCC 实现 | 差异说明 |
|---------|----------|---------|---------|
| **车端发现** | BLE 广播 + 配对码 | BLE 广播 + NFC 触碰 | CCC 支持 NFC 发现更便捷 |
| **身份认证** | SM2 数字签名 | ECDSA (P-256) 签名 | 算法不同，原理相同 |
| **密钥协商** | ECDH + SM2 | ECDH + ECDSA | 密钥交换机制相同 |
| **通信加密** | SM4-CTR/GCM | AES-256-GCM | 算法不同，长度相同 |
| **完整性保护** | SM3 HMAC | HMAC-SHA256 | 算法不同 |
| **测距保护** | UWB ToF + 时间窗口 | UWB Secure Ranging | 机制一致 |
| **证书格式** | 国密证书（SM2） | X.509 v3（ECDSA） | 格式不同，需双向适配 |
| **钥匙分享** | 云端分享 + 协议内分享 | Credential Sharing | CCC 有标准化分享协议 |
| **离线支持** | NFC 刷卡离线 | NFC 刷卡离线 | 均支持 |
| **OTA 更新** | 云端 OTA 通道 | CCC Vehicle Interface | 接口定义不同 |
| **吊销机制** | CRL + 云端即时吊销 | VRL + 即时吊销 | 均支持 |
| **安全等级** | EAL5+（国密认证） | EAL5+（CC 认证） | 认证标准不同 |
| **合规区域** | 中国大陆 | 全球 | 适用地区不同 |

### 5.3 双协议兼容实现策略

为实现同时支持 ICCE 和 CCC，系统采用以下架构：

```
                    ┌──────────────────┐
                    │   应用层统一接口   │（App SDK 统一抽象）
                    └────────┬─────────┘
             ┌───────────────┴───────────────┐
             ▼                                ▼
    ┌─────────────────┐              ┌─────────────────┐
    │   ICCE 协议栈   │              │   CCC 协议栈     │
    │  (SM2/SM3/SM4) │              │(ECDSA/AES/P-256)│
    └────────┬────────┘              └────────┬────────┘
             │                                │
    ┌────────┴────────┐              ┌────────┴────────┐
    │  ICCE 适配层    │              │  CCC 适配层      │
    │  (协议转换)     │              │  (协议转换)      │
    └────────┬────────┘              └────────┬────────┘
             │                                │
    ┌────────┴────────────────────────────────┴────────┐
    │              统一通信抽象层                      │
    │    UWB HAL / BLE HAL / NFC HAL                 │
    └─────────────────────────────────────────────────┘
```

**兼容性策略：**
1. **协议自动识别：** 车端固件同时包含 ICCE 和 CCC 协议栈，配对时自动协商
2. **双证书支持：** 云端同时管理 ICCE 国密证书和 CCC X.509 证书
3. **密钥双签：** 同一把钥匙可同时持有 ICCE 和 CCC 签名版本
4. **用户无感知切换：** App 根据车辆 VIN 识别协议类型，自动选用对应协议

| 防中继攻击 | UWB Secure Ranging | IEEE 802.15.4z |
| 防重放攻击 | 一次性随机数（Nonce） | ICCE / CCC 规范 |
| 会话管理 | Access Token ≤ 1h | OAuth 2.0 最佳实践 |
| 敏感数据 | 端到端加密，密文传输 | GDPR / 数据安全法 |
| 审计日志 | 保留 ≥ 3 年 | 合规要求 |

## 7. 技术约束与风险

### 7.1 技术约束

| 约束类别 | 具体约束 | 影响说明 |
|---------|---------|---------|
| **硬件依赖** | 车辆必须配备 UWB/BLE/NFC 硬件模块 | 老旧车型无法支持数字钥匙 |
| **手机能力** | 手机需支持 SE/eSE 或 TEE | 部分低端手机不支持安全存储 |
| **频段许可** | UWB 频段各国监管政策不同 | 海外部署需申请当地许可 |
| **标准演进** | ICCE/CCC 规范持续更新 | 需持续跟进协议演进 |
| **OTA 通道** | 依赖车厂 OTA 平台能力 | 配对/更新时效受制于 OTA 稳定性 |
| **车企对接** | 每家车企接口规范不同 | 多车型适配工作量较大 |
| **安全认证** | CCC 认证周期长（6-12 个月） | 影响产品上市时间 |
| **数据合规** | 中国《数据安全法》《个人信息保护法》 | 需做数据本地化处理 |
| **国际合规** | GDPR（欧洲）、CCPA（美国） | 海外市场需独立合规审查 |
| **生态壁垒** | Apple CarKey 生态相对封闭 | iOS 端集成受限 |

### 7.2 风险评估矩阵

| 风险编号 | 风险描述 | 可能性 | 影响度 | 风险等级 | 应对策略 |
|---------|---------|-------|-------|---------|---------|
| R-01 | ICCE/CCC 双协议同时支持开发难度大，周期超预期 | 高 | 高 | 🔴 高 | 优先级实现单协议 MVP，再迭代双协议；引入专业协议栈团队 |
| R-02 | 车端 UWB 模组性能不达标（测距精度、功耗） | 中 | 高 | 🔴 高 | 早期与硬件供应商联合定义规格；建立车端性能测试实验室 |
| R-03 | SE 安全芯片供应受限（缺货/替代） | 中 | 高 | 🔴 高 | 引入 2-3 家芯片厂商备选；提前 6 个月锁定库存 |
| R-04 | 云端密钥管理系统（KMS）性能瓶颈 | 低 | 高 | 🟡 中 | 采用分布式 KMS 架构；压力测试验证峰值性能 |
| R-05 | 不同车企 OTA 通道规范差异大 | 高 | 中 | 🟡 中 | 建立标准化 OTA 抽象层；与车厂提前对齐接口 |
| R-06 | 用户隐私数据跨境传输合规问题 | 中 | 高 | 🔴 高 | 数据本地化存储；建立隐私影响评估（PIA）机制 |
| R-07 | 中继攻击技术持续演进，现有防护被突破 | 低 | 高 | 🟡 中 | 建立安全威胁情报机制；预留固件热更新能力 |
| R-08 | App 在各手机品牌兼容性测试覆盖不足 | 中 | 中 | 🟡 中 | 建立主流机型真机测试矩阵；引入云测平台覆盖更多机型 |
| R-09 | CCC 国际认证进度影响海外上市 | 高 | 中 | 🟡 中 | 提前启动认证流程；与认证机构保持密切沟通 |
| R-10 | 项目团队缺乏 ICCE/CCC 协议专业人才 | 中 | 高 | 🔴 高 | 外部引进 + 内部培训；与协议组织建立技术合作 |
