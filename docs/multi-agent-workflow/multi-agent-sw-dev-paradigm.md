## 一、整体架构

本模板定义了一套完整的 multi-agent 软件开发工程流水线，覆盖从需求接收→任务拆解→架构设计→编码实现→代码审查→测试验证→代码合并→交付汇报的全生命周期。

## 二、流水线阶段详解

### 阶段 0：需求接收（教主指令）

> **触发者**：项目总指挥（main / 无影）

项目总指挥接收教主指令后，执行以下动作：

1. 解析教主需求
2. 向 `pm_product` 下发产品需求设计任务
3. 向 `pm_project` 下发任务拆解排期任务
4. 同步向 `arch` 告知项目背景（可并行）

**通信示例：**

```
【项目流水线消息】
项目：<项目名>
阶段：需求接收
来源：main
类型：任务下发
内容：教主发布了新项目需求，请 pm_product 和 pm_project 立即启动工作。
文件位置：<需求原始描述文件>
```

---

### 阶段 1：需求分析与产品设计

> **执行者**：`pm_product`

**职责要点：**
- 深入理解用户需求和业务目标
- 设计完整的产品功能和用户体验
- 编写清晰的技术需求规格说明书（含功能描述、用户故事、验收标准、技术约束）
- 与项目经理和架构师协作确保需求可实现

**输出文档：** 产品需求规格说明书（PRD）

---

### 阶段 2：任务拆解与排期

> **执行者**：`pm_project`

**职责要点：**
- 理解业务需求和技术约束
- 将大任务拆解为可执行的小任务
- 评估每个任务的工作量和依赖关系
- 制定合理的排期计划
- 与产品经理、架构师协调确保计划可行性

**输出文档：** 项目计划书（含 WBS、里程碑、时间表）

---

### 阶段 3：架构设计

> **执行者**：`arch`

**职责要点：**
- 根据产品需求设计系统整体架构
- 选择合适的技术栈和框架
- 定义模块划分和接口规范
- 考虑可扩展性、可维护性和性能要求

**输出文档：** 架构设计文档，必须包含：
- 系统架构图
- 模块职责说明
- 数据流图
- 接口定义（API Contract）
- 部署方案

---

### 阶段 4：代码开发

> **协调者**：`code_designer`  
> **执行者**：`frontend` / `backend` / `embedded`（通过 `sessions_spawn` 创建子 agent）

#### 4.1 code_designer 工作流

```
1. 接收架构设计文档
2. 拆解为具体开发任务（分配给 frontend/backend/embedded）
3. 生成完整的接口文档（API Contract）
4. 协调三方开发进度，确保接口一致性
5. 监控开发进度，阻塞时及时调整
```

#### 4.2 子 Agent 职责

**frontend：**
- 使用指定技术栈（React/Vue 等）开发前端组件
- 实现 UI 设计和用户体验
- 与后端 API 对接联调
- 性能优化与单元测试

**backend：**
- 使用指定技术栈（Node.js/Python/Go 等）开发后端服务
- 实现 API 接口和业务逻辑
- 数据存储与数据库操作
- 系统安全性和性能保障

**embedded：**
- 使用指定嵌入式平台（ARM/NXP 等）进行开发
- 实现硬件驱动和底层通信协议
- 优化内存使用和功耗管理
- 确保实时性和稳定性

---

### 阶段 5：代码审查

> **执行者**：`code_reviewer`

**职责要点：**
- 检查代码是否符合架构设计和接口规范
- 验证代码质量、性能和安全性
- 确保代码风格一致性和最佳实践
- 发现潜在的 bug 和逻辑问题
- 提供具体的改进建议

**审查结果格式：**

```
【项目流水线消息】
项目：<项目名>
阶段：代码审查
来源：code_reviewer
类型：审查通过 | 审查不通过
内容：<详细说明原因 + 具体修改建议>
文件位置：<被审查的代码文件路径>
```

---

### 阶段 6：测试

> **协调者**：`test`  
> **执行者**：`frontend_test` / `backend_test` / `embedded_test`

#### 6.1 test 工作流

```
1. 接收需求文档和代码设计
2. 制定测试策略
3. 协调三个子 agent 执行测试
4. 验证功能完整性、边界情况、异常处理
5. 跟踪 bug 修复和回归测试
6. 输出测试报告
```

#### 6.2 子测试 Agent 职责

**frontend_test：**
- UI 组件功能与交互测试
- 前端与后端 API 集成验证
- 错误处理与边界情况
- 跨浏览器/设备兼容性

**backend_test：**
- 所有 API 接口功能与性能测试
- 数据处理和业务逻辑验证
- 安全性检查（SQL 注入、XSS 等）
- 高并发与压力测试

**embedded_test：**
- 硬件驱动和底层功能测试
- 通信协议正确性验证
- 功耗和性能指标检测
- 极端环境稳定性测试

**测试报告格式：**

```
【项目流水线消息】
项目：<项目名>
阶段：测试
来源：test
类型：测试通过 | 测试不通过
内容：<测试覆盖率、Bug 统计、关键问题>
文件位置：<测试报告文件路径>
```

---

### 阶段 7：代码合并与交付

> **执行者**：`repo_manager`

**职责要点：**
- 验证代码审查和测试都已通过（前置条件校验）
- 执行代码合并到主分支
- 管理版本标签和发布流程
- 维护代码库的整洁和一致性
- 向项目总指挥汇报交付结果

**合并条件（全部满足方可合并）：**
- [ ] 代码审查通过
- [ ] 所有测试用例通过
- [ ] 无未解决的阻塞性问题

**合并完成消息：**

```
【项目流水线消息】
项目：<项目名>
阶段：代码合并
来源：repo_manager
类型：合并完成
内容：<版本号、变更摘要、交付物清单>
文件位置：<合并后的代码仓库路径>
```

---

### 阶段 8：最终交付汇报

> **执行者**：`main`（项目总指挥）

项目总指挥汇总所有阶段的产出，向教主汇报最终交付结果。

## 三、通信协议规范

所有 agents 必须统一使用以下格式进行通信：

```
【项目流水线消息】
项目：<项目名>
阶段：<当前阶段>
来源：<发送方Agent ID>
类型：<消息类型>
内容：<具体描述>
文件位置：<共享工作区相关文件路径>
```

### 消息类型枚举

| 类型 | 说明 | 触发场景 |
|------|------|----------|
| 任务下发 | 上游向下游分配具体任务 | pm→arch, arch→code_designer 等 |
| 审查通过 | 代码审查结果为通过 | code_reviewer |
| 审查不通过 | 代码审查未通过，需修改 | code_reviewer |
| 测试通过 | 测试阶段全部通过 | test |
| 测试不通过 | 测试发现问题，需修复 | test |
| 合并完成 | 代码已合并到主分支 | repo_manager |
| 进度汇报 | 阶段性进度同步 | 任意 agent |
| 阻塞上报 | 遇到无法自行解决的阻塞 | 任意 agent |

## 四、实施建议

### 4.1 Multi-Agent 协作模式

- **串行依赖阶段**：需求 → 架构 → 开发 → 审查 → 测试 → 合并（严格顺序）
- **可并行阶段**：frontend/backend/embedded 开发可并行推进，通过 code_designer 协调
- **质量门禁**：每个阶段设置明确的交付物标准，不满足不进入下一阶段

### 4.2 阻塞处理机制

任何 agent 遇到无法自行解决的问题，应立即通过通信协议向 main（项目总指挥）上报，由总指挥协调资源解决，避免流水线卡死。

### 4.3 教主汇报原则

- 阻塞情况：第一时间实时汇报
- 阶段完成：汇总产出后汇报
- 交付完成：完整交付物清单 + 版本信息

```
                │
                ▼
        code_designer ──► 接口文档
        /      |       \
        ▼      ▼        ▼
    frontend  backend  embedded （并行开发）
        │      │        │
        ▼      ▼        ▼
    frontend_ backend_ embedded_test
    test     test
        \      |        /
         ▼     ▼        ▼
        code_reviewer ← 审查
              │
              ▼
            test ← 协调三方测试
              │
              ▼
        repo_manager ← 合并
              │
              ▼
        main ← 向教主汇报交付
```
